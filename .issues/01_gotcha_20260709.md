# Gotcha register — cedar-vs-spicedb benchmark

Non-obvious traps that remain TRUE for the current codebase. Status lifecycle:
`open → acknowledged → mitigated → resolved` (or `wont-fix`). Consolidated register — resolved
historical findings live in git history; lasting lessons are folded into `docs/` where they belong.

---

## G1 — SpiceDB bulk import (binary COPY) breaks against Postgres 18
- **Severity:** high | **Status:** mitigated (by design choice)
- **Location:** SpiceDB v1.54.0 `ImportBulkRelationships` vs `postgres:18.4` — fails mid-stream with
  `FATAL: terminating connection because protocol synchronization was lost (SQLSTATE 08P01)` during
  `copy "relation_tuple" … from stdin binary`.
- **Why it's a gotcha:** the recommended fast bulk path fails only at the wire-protocol level, mid
  transaction, with an opaque error — small imports may pass, so it looks intermittent.
- **Mitigation:** the seeder uses `WriteRelationships` + `OPERATION_TOUCH` in 1000-record batches
  (the server's default cap; idempotent re-runs). Revisit COPY when SpiceDB documents PG18 support.

## G2 — SpiceDB object IDs forbid `:` (colon)
- **Severity:** medium | **Status:** resolved (convention locked in)
- **Location:** object-ID regex `^(([a-zA-Z0-9/_|\-=+]{1,})|\*)$`.
- **Why it's a gotcha:** URN-style IDs with `:` are rejected only at write time, per update, after
  all the schema plumbing already works. Registry IDs use `/` (`ep/finance/invoice-approve`) —
  keep it that way (documented in `catalog/services.json`).

## G3 — search_path isolation is a Postgres mechanism, not a SpiceDB feature
- **Severity:** medium | **Status:** acknowledged (works; re-verify on upgrades)
- **Location:** `db/bootstrap.sh` (`ALTER ROLE spicedb IN DATABASE authz SET search_path=spicedb`).
- **Why it's a gotcha:** SpiceDB docs never mention schema selection; isolation works because
  SpiceDB issues unqualified table names through pgx. A future release could schema-qualify and
  silently land tables in `public`.
- **Action on every SpiceDB upgrade:** re-run the smoke test — SpiceDB's tables must appear in
  schema `spicedb` only (`public`/`cedar` must gain none). Fallback: separate databases.

## G4 — Cross-scale/seed reseeding requires -wipe (IDs overlap)
- **Severity:** high | **Status:** mitigated (hard guard in place)
- **Location:** `cmd/authz-seed` + `internal/seed/writer_postgres.go` (seed_checkpoints.params).
- **Why it's a gotcha:** IDs are deterministic (`org-00001`, …), so datasets of different
  seeds/scales OVERLAP; Postgres `ON CONFLICT DO NOTHING` keeps stale rows while SpiceDB `TOUCH`
  overwrites — the engines would silently diverge and only the equivalence gate (far from the
  cause) would notice.
- **Mitigation:** checkpoints carry a `seed/scale/budget-signature` fingerprint; resuming under
  different params hard-fails demanding `-wipe`; `-wipe` truncates strictly (only
  undefined_table tolerated), deletes SpiceDB relationships per definition, and removes stale
  ground-truth tuple files.

## G5 — SpiceDB relation_tuple is MVCC: naive counts over-count
- **Severity:** medium | **Status:** acknowledged (query pattern documented)
- **Location:** schema `spicedb.relation_tuple` (deleted rows persist as tombstones until GC).
- **Why it's a gotcha:** `SELECT count(*)` counts logically-deleted tuples for up to the GC window
  (~24h) — after a wipe/reseed the number looks wrong and can trigger false "stale data" alarms.
- **Action:** always filter live rows: `WHERE deleted_xid = '9223372036854775807'::xid8`.

## G6 — ABAC/PBAC principal facts travel as check-time context for SpiceDB
- **Severity:** low (inherent architectural asymmetry) | **Status:** acknowledged (documented)
- **Location:** `schema/spicedb/schema.zed` caveats `doc_attrs` / `po_limits`; bench tuples carry
  `clearance`/`principal_division`/`principal_region` (+ `amount`/`region` for PBAC).
- **Why it's a gotcha:** SpiceDB cannot look up arbitrary subject attributes server-side — the
  caller supplies them, so SpiceDB's measured latency EXCLUDES the principal-attribute fetch that
  Cedar's end-to-end number INCLUDES. Under-supplying context yields `CONDITIONAL`, which this repo
  treats as an error everywhere.
- **Action:** deny tuples must always use the persona's TRUE attributes (never fabricated context),
  or the engines judge different facts. See docs/03 fairness notes.

## G7 — minimize_latency numbers are a cache-hit best case
- **Severity:** low | **Status:** acknowledged (documented in docs/03)
- **Location:** benchmark cells replay a fixed tuple working set for ~10s; SpiceDB's quantized
  cache (~5s window) then serves near-100% hits.
- **Why it's a gotcha:** quoting `minimize_latency` as "SpiceDB production performance" overstates
  it for workloads spread across millions of resources or with interleaved writes;
  `fully_consistent` bounds the miss side.

## G8 — Benchmark harnesses lie easily (client-side artifacts)
- **Severity:** high (as a class) | **Status:** mitigated (fairness rules encoded in the harness)
- **Location:** `internal/bench/bench.go`, `internal/config/config.go`.
- **Why it's a gotcha:** three defaults silently corrupted early measurements: pgxpool's
  `MaxConns = NumCPU` sat BELOW the bench concurrency (measured pool queueing, not the engine);
  single-threaded warmup never opened the pool the timed section used; errored checks were timed
  (fast failures flatter an engine). None of these announce themselves.
- **Mitigation (permanent harness rules):** `pool_max_conns` ≥ max concurrency; warmup at the
  cell's own concurrency covering the full tuple set; errored checks excluded from latency AND
  throughput; preallocated reservoir sampling keeps GC out of the timed window; `cedar-eval` times
  only `cedar.Authorize` via `Prepare`/`EvaluatePrepared`. Any harness change must re-justify
  these against docs/03 §fairness.

## G9 — Weak local-dev default credentials
- **Severity:** medium | **Status:** open (fine locally, unsafe anywhere shared)
- **Location:** `internal/config/config.go`, `docker-compose.yml`, `db/bootstrap.sh` — roles
  `cedar`/`spicedb` default to their own names as passwords; SpiceDB preshared key defaults to
  `benchkey`.
- **Suggested mitigation:** require credentials (fail fast) outside local dev; never deploy the
  compose file as-is.

## G10 — Container images pinned to tags, not digests
- **Severity:** low | **Status:** open (Renovate `pinDigests` will manage on PRs)
- **Location:** `docker-compose.yml` (`postgres:18.4`, `authzed/spicedb:v1.54.0`), `Dockerfile`
  (`golang:1.26-bookworm`, distroless).
- **Why it's a gotcha:** numeric tags still move (patch rebuilds); reproducibility needs
  `@sha256:` digests.

## G11 — The facade has no machine-checkable API contract
- **Severity:** low | **Status:** open
- **Location:** `POST /v1/authorize` exists only as code + `http/*.http` examples.
- **Suggested mitigation:** generate an OpenAPI spec into `.claude/docs/` wired by a path-scoped
  rule; run contract checks against the handler.

## G12 — Host-run tools need a host-reachable SpiceDB endpoint
- **Severity:** medium | **Status:** mitigated (Makefile override)
- **Location:** `internal/config/config.go` (`SPICEDB_ENDPOINT` default `spicedb:50051`),
  `docker-compose.yml` (sets `spicedb:50051` for the containerized facade), `Makefile`.
- **Why it's a gotcha:** `spicedb:50051` is a Docker-network name. The facade *inside* Compose
  resolves it, but `make seed`/`verify`/`bench`/`run` execute on the HOST via `go run`, where
  `spicedb` has no DNS record — the gRPC client fails every call with `code = Unavailable desc =
  name resolver error: produced zero addresses`. It looks like an equivalence/engine failure
  (the whole `spicedb-fully_consistent` column errors) but Cedar passes, so the real cause —
  a name that resolves in one context and not the other — is easy to misread.
- **Mitigation:** the `Makefile` `ENV` prelude forces `SPICEDB_ENDPOINT=localhost:50051` for all
  host targets (the gRPC port Compose maps to the host), overridable via `SPICEDB_HOST_ENDPOINT`.
  Rule of thumb: containers use `spicedb:50051`, the host uses `localhost:50051`.
