## G1 — No automated tests on a security-critical authorization service
- **Severity:** critical | **Status:** mitigated
- **Location:** originally the whole repo; the 8 scenarios in `demo/cedar.http` were only manual.
- **Why it's a gotcha:** this service decides allow/deny. Any change to `policies/policy.cedar` or
  the entity-building code can silently flip a decision (an authz bypass) with nothing to catch it.
- **Impact:** undetected authorization regressions reach production.
- **Suggested mitigation:** encode the 8 cases (+ validation and error paths) as automated tests.
- **2026-07-09 update:** the 8 scenarios are now pinned in
  `internal/adapter/outbound/cedar/engine_test.go` (`TestDecisionMatrix`, `make test`), and HTTP
  validation is covered in `internal/adapter/inbound/rest/handler_test.go`. **Remaining:** the
  Postgres adapter has no unit test (only end-to-end coverage) — keep open as a sub-item.

## G2 — No linter/formatter configured
- **Severity:** low | **Status:** mitigated
- **Location:** originally no ESLint/Prettier; `package.json` had only a `start` script.
- **Why it's a gotcha:** style drift; the `format.sh` hook was idle, giving a false sense of
  "auto-format on".
- **2026-07-09 update:** Go ships `gofmt` + `go vet`, both wired via the Makefile (`make fmt`,
  `make vet`) and `format.sh` now gofmts `.go` files on save. ESLint/Prettier no longer relevant.
  Consider adding `golangci-lint` for deeper linting (optional).

## G4 — Container base images use floating tags
- **Severity:** medium | **Status:** open
- **Location:** `Dockerfile` → `golang:1.26-bookworm` (build) and
  `gcr.io/distroless/static-debian12:nonroot` (runtime).
- **Why it's a gotcha:** these tags move over time, so builds are not fully reproducible and a base
  change can alter/break the image without any repo change — the "pin numeric + digest" rule.
- **2026-07-09 update:** the earlier `node:22-slim` finding now applies to the Go build/runtime
  images. Pin to `@sha256:<digest>`; Renovate `pinDigests` (in `renovate.json`) will manage digests.

## G9 — Weak default database credential
- **Severity:** medium | **Status:** open
- **Location:** `internal/config/config.go` (`DB_PASSWORD` defaults to `"app"`) and the compose
  default `${DB_PASSWORD:-app}`.
- **Why it's a gotcha:** if the env is unset, the service silently runs with a trivial, well-known
  password instead of failing fast.
- **2026-07-09 update:** carried over to the Go config unchanged (kept for local-dev parity).
  Suggested: require the credential (fail fast) outside local dev.

## G10 — Schema changes need a fresh volume (no migration tool)
- **Severity:** medium | **Status:** open
- **Location:** `db/init.sql` (runs only via `docker-entrypoint-initdb.d` on an empty volume).
- **Why it's a gotcha:** editing `init.sql` has no effect on an existing database; applying it means
  `make reset` / `docker compose down -v` (data loss), easy to do accidentally.
- **2026-07-09 update:** unchanged by the rewrite. Suggested: adopt `golang-migrate`; keep
  `init.sql` for first-boot seed only.
