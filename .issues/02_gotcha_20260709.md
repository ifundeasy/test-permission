# Gotcha register — 2026-07-09 (benchmark pivot: Cedar vs SpiceDB)

New findings discovered while building the dual-engine benchmark. Prior findings live in
`01_gotcha_20260709.md` (statuses updated there). Status lifecycle:
`open → acknowledged → mitigated → resolved` (or `wont-fix`).

---

## G11 — SpiceDB bulk import (binary COPY) breaks against Postgres 18
- **Severity:** high | **Status:** mitigated
- **Location:** SpiceDB v1.54.0 `ImportBulkRelationships` vs `postgres:18.4`; observed as
  `FATAL: terminating connection because protocol synchronization was lost (SQLSTATE 08P01)` during
  `copy "relation_tuple" … from stdin binary` (see postgres/spicedb logs, 2026-07-08 20:00 UTC).
- **Why it's a gotcha:** the recommended fast bulk path fails only at the wire-protocol level, mid
  transaction, with an opaque error — small imports may pass, so it looks intermittent. Postgres 18
  introduced protocol changes; SpiceDB's binary COPY appears incompatible (a small first batch of 35
  rels succeeded; validation errors are NOT the cause — invalid IDs produce clean InvalidArgument).
- **Impact:** seeding aborts mid-phase; naive retries burn time.
- **Mitigation applied:** seeder uses `WriteRelationships` + `OPERATION_TOUCH` in 1000-record
  batches (server default cap is exactly 1000). ~16k rels/s observed — acceptable. Revisit COPY
  path when SpiceDB documents Postgres 18 support.

## G12 — SpiceDB object IDs forbid `:` (colon)
- **Severity:** medium | **Status:** resolved
- **Location:** object-ID regex `^(([a-zA-Z0-9/_|\-=+]{1,})|\*)$`; original registry IDs were
  `ep:finance:invoice-approve`.
- **Why it's a gotcha:** ID conventions that work everywhere else (URNs with `:`) are rejected only
  at write time, per-update, after schema/write plumbing already works.
- **Mitigation applied:** registry IDs use `/` separators (`ep/finance/invoice-approve`);
  documented in `catalog/services.json`.

## G13 — Cross-scale reseeding silently diverges the engines
- **Severity:** high | **Status:** mitigated
- **Location:** `internal/seed` — IDs are deterministic (`org-00001`, …) so miniature and full
  datasets OVERLAP; Postgres `ON CONFLICT DO NOTHING` keeps the stale row while SpiceDB `TOUCH`
  overwrites (and stale extra relationships survive).
- **Why it's a gotcha:** both engines report successful seeding; only the equivalence gate would
  catch the divergence, far from the cause.
- **Mitigation applied:** `authz-seed -wipe` (truncates cedar tables + DeleteRelationships per
  definition + clears checkpoints); documented as REQUIRED when changing `-scale`.

## G14 — SpiceDB schema isolation via search_path is not an official feature
- **Severity:** medium | **Status:** acknowledged (works, by design of Postgres not SpiceDB)
- **Location:** `db/bootstrap.sh` (`ALTER ROLE spicedb IN DATABASE authz SET search_path=spicedb`).
- **Why it's a gotcha:** SpiceDB docs never mention schema selection; this relies on SpiceDB issuing
  unqualified table names through pgx. A future SpiceDB release could schema-qualify names and land
  tables in `public`.
- **Impact:** silent co-location with app tables after an upgrade.
- **Suggested mitigation:** smoke test after every SpiceDB upgrade (`\dt spicedb.*` must show its 9
  tables; `public`/`cedar` must not gain any). Fallback: separate databases on the same server.

## G15 — ABAC principal attributes travel as check-time context for SpiceDB
- **Severity:** low (documented architectural asymmetry) | **Status:** acknowledged
- **Location:** `schema.zed` caveat `doc_attrs`; bench tuples carry `clearance`/`principal_division`.
- **Why it's a gotcha:** SpiceDB cannot look up arbitrary subject attributes server-side, so the
  caller must supply them — meaning SpiceDB's measured latency EXCLUDES the principal-attribute
  fetch that Cedar's end-to-end number INCLUDES. Apple-to-apple requires reading the fairness notes.
- **Mitigation:** documented in README (fairness) and mirrored by the `cedar-eval` cell (engine-only
  cost); deny tuples always use the persona's TRUE attributes so both engines judge the same facts.

## G16 — Bench harness measured client-side artifacts, not the engines (found by adversarial review)
- **Severity:** high | **Status:** resolved (2026-07-09, same day)
- **Location:** `internal/bench/bench.go`, `internal/config/config.go` (pgx pool DSN).
- **Why it's a gotcha:** three compounding, invisible-by-default harness biases: (1) pgxpool defaults
  to `MaxConns = NumCPU` (16 here) — the c=32 Cedar cells measured pool-acquire queueing while
  SpiceDB multiplexed one warmed gRPC conn; (2) warmup was single-threaded (never opened the pool
  capacity the timed section used) and covered only half the tuple set; (3) errored checks were
  timed and counted (fast failures would flatter an engine), and unbounded latency slices caused
  mid-measurement realloc/GC noise in the tails of >1M-checks/s cells.
- **Fix applied:** explicit `pool_max_conns=48&pool_min_conns=8` (≥ max concurrency); warmup at the
  cell's own concurrency covering the full tuple set; errored checks excluded from latency AND
  throughput; per-worker preallocated reservoir sampling (exact mean/max, sampled percentiles);
  `cedar-eval` now times ONLY `cedar.Authorize` via Prepare/EvaluatePrepared (no per-call
  conversion overhead); eval-only cache is index-aligned (no key collisions on context-only
  differences). README fairness notes updated; first-run results superseded.

## G17 — Seeder resume could silently mix datasets across seeds/scales
- **Severity:** high | **Status:** resolved (2026-07-09)
- **Location:** `cmd/authz-seed/main.go`, `internal/seed/writer_postgres.go` (seed_checkpoints).
- **Why it's a gotcha:** checkpoints recorded only (engine, phase) — rerunning with a different
  `-seed`/`-scale` and `-resume` (the default) skipped seeding entirely while the tuples phase
  happily rewrote ground truth for the NEW flags; `wipeAll` also swallowed every TRUNCATE error and
  left tuple files behind, so "[wipe] done" could lie.
- **Fix applied:** checkpoints now store a `params` fingerprint (`seed=… scale=…`); resuming under
  different params hard-fails demanding `-wipe`; wipe only tolerates undefined_table (42P01),
  aborts on any other TRUNCATE error, and removes stale `bench/tuples/*.json`.

## G18 — Facade returned 400 (with internal detail) for backend faults
- **Severity:** medium | **Status:** resolved (2026-07-09)
- **Location:** `internal/adapter/inbound/rest/handler.go`, `internal/core/service/router.go`.
- **Why it's a gotcha:** Postgres/SpiceDB outages surfaced as `400 Bad Request` with raw driver/gRPC
  error text — misclassifying server faults as client errors (breaks monitors/load tools) and
  leaking internals.
- **Fix applied:** `service.ErrBadRequest` sentinel → 400 only for caller mistakes; backend faults
  → `502 {"error":"authorization backend failed"}` with detail server-side only. Also:
  check-time integers ≥ 2^53 are now rejected (structpb float64 precision — G15's cousin).
